Java 业务层

Java 现在的职责已经收得比较清楚：不再自己轮询基金净值，而是专注做业务确认、持仓回算、对前端聚合输出。

fund holding sip watchlist overview ops
前端请求进入 Java API 统一鉴权、统一响应格式、统一屏蔽底层是 MySQL、Redis 还是 Python bridge。
消费基金 ready 消息 先确认 pending 的手工买卖和定投记录，再按基金聚合，避免重复回算。
统一回算持仓 最终仍通过 rebuildSnapshotsFrom 落到日终快照，并刷新或清理当前态缓存。

Python 数据侧

Python 现在是基金市场数据的唯一采集入口。它负责抓 AKShare、补偿晚到净值、写 outbox，并且给 30 天外历史走势提供 bridge。

sync-basics nav/seed poll-evening poll-retry outbox/publish cleanup
09:00 seed 从 funds 生成当天待抓基金集合，写入 fund_nav_pending_keys，不往净值事实表里插空值占位。
18:15 起轮询净值 晚间每 2 小时轮询一次当天未完成基金；次日 07:30 和 09:00 继续补偿前一交易日没拿到的净值。
成功后写库并发消息 upsert fund_nav_history / growth，再写 outbox，最终批量发 fund.nav.ready.batch。

数据层与日间动态态

MySQL 负责正式事实，Redis 负责白天展示态。现在已经不是“latest 表直读”，而是“日终结果 + 日间缓存”的组合。

holding_daily_snapshots holding_operation_records fund_nav_history (30天) fund_nav_growth (30天) Redis intraday state
09:00 预热 Redis 把用户前一日持仓快照加载到 Redis，作为当天 intraday state 的基线。
白天读取优先走 Redis 前端展示先读 Redis 日间态；后续如果接入实时估值，就在这里更新 currentNav 和盈亏。
晚间正式净值确认后清缓存 当某基金完成 handleConfirmedNav，Redis 对应 intraday state 删除，次日重新以正式快照为基线加载。

还没实现的部分

这几块已经在结构里预留了入口，但还没有真正跑成生产链路。

  • RocketMQ 仍是消息模型和消费入口已就位，但 Python publisher 还是 log 模式，没有真正接 broker。
  • Airflow DAG 还没写成正式调度文件，当前只有 ops API 和任务步骤说明。
  • 实时估值驱动的 intraday 刷新还没闭环，EstimatePollingJob 目前只是估值轮询入口。
  • Python 的 .env 还没有自动加载，本地运行仍主要依赖 Run Configuration 或 export 环境变量。
